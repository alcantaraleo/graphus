# Conventions

## Language & Build

- Java 21, Gradle with Kotlin DSL (`build.gradle.kts`)
- Shared build conventions live in `buildSrc/src/main/kotlin/graphus.java-conventions.gradle.kts`
- Version is sourced from `version.txt` at the repo root
- Publishing targets GitHub Packages (`GITHUB_REPOSITORY`, `GITHUB_TOKEN` env vars)

## Code Style

- No star imports — always use fully qualified imports
- Constructor injection only; no field injection (`@Autowired` on fields)
- `final` classes and fields wherever mutation is not needed
- Package-private static nested classes for subcommands (see `GraphusCommand`)
- No Spring Boot runtime — the CLI is a plain Java application using Picocli

## Testing

- JUnit 5 (`junit-bom:5.11.4`) with JUnit Jupiter
- Mockito for mocking
- Tests live under `src/test/java` in each module

## Module Dependencies

```
graphus-model   ← no internal deps
graphus-parser  ← graphus-model
graphus-indexer ← graphus-model
graphus-cli     ← graphus-model, graphus-parser, graphus-indexer
```

No circular dependencies between modules. `graphus-parser` and `graphus-indexer` are independent of each other.

## Naming

- Symbol IDs follow the pattern: `fully.qualified.ClassName.methodName(param.Type)` — generated by `SymbolIdResolver`
- Kotlin top-level functions are emitted under a synthetic `<package>.<File>Kt` facade class so their IDs match the JVM bytecode shape that other tools (and Java callers) see
- Kotlin properties are emitted as `FieldNode`s with `val ` or `var ` prefixed onto the type string to preserve the source mutability semantics
- Logical collection names are user-defined via `--collection`; Graphus maps them to Chroma collections or normalized SQLite table identifiers
- The state directory defaults to `{repo}/.graphus/` and stores `checksums.json` (covering `.java` and `.kt` files), `config.json`, and optionally `graphus.db`

## Source layout & language detection

- `ModuleDescriptor` carries two parallel root lists: `sourceRoots` for Java and `kotlinSourceRoots` for Kotlin. At least one of the two must be non-empty.
- `RepositoryLayoutDetector` automatically discovers `src/main/java` and `src/main/kotlin` per module for both Gradle and Maven layouts, and emits Kotlin-only `ModuleDescriptor`s when no Java root exists.
- The CLI `--source` flag accepts paths to either Java or Kotlin source roots; `CliWorkspaceLayouts` classifies each path heuristically (`/kotlin/` segments → Kotlin) and routes them to the correct list on `ModuleDescriptor`.
- `ProjectParser.createParserConfiguration` only registers existing directories with `JavaParserTypeSolver`, so Kotlin-only modules no longer crash with `IllegalStateException`.

## Cross-language call edges

- Per-language call-graph builders (`CallGraphBuilder`, `KotlinCallGraphBuilder`) emit structured `UnresolvedCallRecord`s instead of relying on string parsing. Each record carries the caller ID, callee simple name, arity, the placeholder `UnresolvedNode` ID, and the origin language.
- `CrossLanguageCallResolver` runs after both per-language passes and replaces placeholder edges with real `Java→Kotlin` / `Kotlin→Java` edges when exactly one `MethodNode` in the other language matches by simple name and arity. Ambiguous matches are left as unresolved on purpose.
- Cross-language edges are explicitly **best-effort**: Kotlin extension functions, lambdas, infix calls, and operator overloads are not yet modeled.

## AI Tool Integration

The `install` command writes tool-specific rule/config files into target projects. Each tool has a dedicated adapter implementing `ToolAdapter`:

- `CursorAdapter` → `.cursor/rules/graphus.mdc`
- `ClaudeCodeAdapter` → equivalent for Claude Code

To add a new tool, implement `ToolAdapter` and register it in `InstallCommand`.
